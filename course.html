01

Welcome to Smallville

A simulated town full of AI characters who remember, plan, gossip — and have opinions about your Valentine's Day party.

What is this, exactly?

This codebase is a Stanford research project that answers a simple question: what if AI characters actually remembered things?

The town is called Smallville. It has a cafe, homes, a school, a park. Up to 25 characters live there — Isabella Rodriguez, Klaus Mueller, Maria Lopez. They wake up, plan their day, go to work, eat lunch, bump into neighbours, gossip about the Valentine's Day party someone's planning at Hobbs Cafe.

None of this is scripted. The characters decide what to do based on their memories, personalities, and the LLM processing their thoughts.

What you actually see when you run it

1
A 2D pixel-art map of Smallville opens in your browser

Built with Phaser.js and a Django server. Characters are little sprite icons you can watch move around.

2
You type run 100 in the terminal

The simulation takes 100 steps. Each step = 10 seconds of in-game time. So 100 steps is about 16 game-minutes.

3
Characters start moving — and talking

When two characters walk close enough, they have a conversation. Both generated in real-time by the LLM. Both stored in memory afterward.

4
You save, exit, reload — and they remember

Tomorrow's Isabella remembers what happened yesterday. That's what makes this different from normal AI chatbots.

The core insight of this project

Regular LLMs have no memory between conversations. Generative Agents solves this by keeping a diary — a database of every event, thought, and conversation — and selectively injecting the most relevant memories into each prompt. Isabella doesn't just know her name. She knows what she did yesterday.

What each component actually does

⚙️
ReverieServer — The conductor

Runs the main while loop. Reads the environment file from the frontend, tells each persona to think, writes the movement file back. It's the thing you talk to when you type run 100.

🗺️
Maze — The world

A 140×100 grid of tiles. Each tile knows its location (sector, arena, game object) and what events are happening there. Isabella standing at the piano = the piano tile has an event "Isabella is playing piano".

🧠
Persona — The agent

Holds three memory systems (long-term, short-term, spatial) and delegates all thinking to the cognitive modules. It's a coordinator, not a thinker.

📔
AssociativeMemory — The diary

Stores every event, thought, and conversation as a node with an embedding. When Isabella needs to decide something, retrieval searches this diary using cosine similarity.

📋
Scratch — The sticky note

The agent's current working state: what am I doing right now? Where am I going? Who am I talking to? Resets or updates every tick. If AssociativeMemory is the diary, Scratch is the note on the fridge.

🌳
SpatialMemory — The mental map

A JSON tree of what the agent knows about the town's layout: world → sector → arena → game objects. Isabella knows her house has a bedroom with a bed. She doesn't need to explore to find it.

🍎
WorldResourceManager — The world's pantry

Andus's addition to the original paper. Tracks real-world resources like food in fridges and coffee at the cafe. Agents develop needs (hunger, energy, social) that decay over time and influence their plans.

03

The Memory That Makes Isabella, Isabella

How the code stores and retrieves memories — and why it's more like a detective than a database.

Every memory is a ConceptNode

The AssociativeMemory class stores memories as ConceptNode objects. Think of each node as an index card in a detective's evidence board.

associative_memory.py 
class ConceptNode:
  def __init__(self, node_id, ...):
    self.type = node_type  # event/thought/chat
    self.subject = s
    self.predicate = p
    self.object = o
    self.description = description
    self.poignancy = poignancy
    self.embedding_key = embedding_key
    self.last_accessed = self.created
Plain English

Each memory has a type — it's either something that happened (event), a thought the agent formed, or a conversation.

Every memory is stored as a triplet: Subject → Predicate → Object. Like "Isabella → is planning → a Valentine's party."

Poignancy is how emotionally significant this memory was, scored 1–10 by the LLM.

The embedding_key is the sentence that got turned into a vector — used for semantic search later.

last_accessed tracks when this memory was last retrieved. Recent access = more likely to surface again.

The retrieval score: a three-way competition

When Isabella needs to make a decision, the code searches her entire memory diary. Not by keyword — by meaning. The retrieval score combines three factors:

⏰
Recency
Memories accessed more recently score higher. Fades exponentially — yesterday gets 0.99×, last week gets 0.90×.
❤️
Poignancy
How emotionally significant the memory was. Scored 1–10 by the LLM when first stored. Big events surface faster.
🔍
Relevance
Cosine similarity between the current event's embedding and the memory's embedding. Semantically similar = high score.
retrieve.py — cosine similarity 
def cos_sim(a, b):
  return dot(a, b) / (norm(a) * norm(b))


def retrieve(persona, perceived):
  retrieved = dict()
  for event in perceived:
    relevant_events = persona.a_mem.retrieve_relevant_events(
      event.subject, event.predicate, event.object)
    relevant_thoughts = persona.a_mem.retrieve_relevant_thoughts(
      event.subject, event.predicate, event.object)
    retrieved[event.description] = {
      "curr_event": event,
      "events": list(relevant_events),
      "thoughts": list(relevant_thoughts)
    }
  return retrieved
Plain English

cos_sim measures how similar two vectors are by the angle between them. Score of 1.0 = identical meaning. Score of 0.0 = completely unrelated.

retrieve loops over what Isabella currently perceives (what's near her on the map).

For each perceived event, it searches the memory diary for the most semantically relevant past events AND thoughts.

Returns a dictionary: current event → what memories are related. This bundle goes into the LLM prompt for decision-making.

Reflection: when memories become wisdom

Occasionally, the agent stops and reflects. It takes its most recent high-poignancy memories and asks the LLM: "What insights can I draw from all this?" The result becomes a new memory node of type thought, at depth 1 (one level above raw events).

💡
This is why agents develop opinions

If Klaus saw Isabella working late three times this week, a reflection might produce: "Klaus Mueller believes Isabella Rodriguez is a very dedicated person." This thought then influences how Klaus talks about Isabella in future conversations — even if he hasn't spoken to her recently. The agent is genuinely building a model of the world.

What the agent notices — The code scans nearby tiles within the agent's vision radius (8 tiles). Any event on those tiles that passes the attention filter gets added to a list of perceived events. For example: Klaus Mueller standing at the piano, or an unmade bed in the bedroom. Unimportant things are filtered out by the attention bandwidth setting.
Which memories are relevant — For each perceived event, the code searches AssociativeMemory using cosine similarity to find the most relevant past events and thoughts. The result is a bundle of context that goes into the LLM prompt.
What to do next — The plan module checks if the current action is still valid. If not (or if it's time to plan the next task), it calls the LLM with the agent's personality, the retrieved memories, and the current game time to decide what to do next. Plans cascade: day plan → hourly plan → current task.
Forming insights — Reflection only triggers when the sum of poignancy scores for recent memories exceeds a threshold (default: 150). When it fires, the LLM synthesises the most important recent memories into higher-level thoughts, which get stored back in memory at depth+1.
Actually moving — Given the planned action's target address (e.g., "the_ville:Hobbs Cafe:cafe:espresso machine"), the code uses A* pathfinding through the collision maze to find the route. The next tile coordinate gets written to the movement file.

The main loop in code

reverie.py — start_server() main loop 
while (True):
  if int_counter == 0: break

  curr_env_file = f"{sim_folder}/environment/{self.step}.json"
  if check_if_file_exists(curr_env_file):
    # Read new agent positions from frontend
    new_env = json.load(open(curr_env_file))

    self.tick_needs()

    # Write movement decisions for frontend
    with open(f"{sim_folder}/movement/{self.step}.json", "w") as f:
      f.write(json.dumps(movements, indent=2))

    self.step += 1
    self.curr_time += timedelta(seconds=self.sec_per_step)
Plain English

Keep looping until we've taken the requested number of steps.

Wait for the frontend to drop a new environment/{step}.json file — this is the frontend saying "I've moved everyone, it's your turn."

For each agent, call their move() function — that's the perceive→retrieve→plan→execute pipeline.

tick_needs() is Andus's addition — decay hunger, energy, social, etc. for every agent.

Write movement/{step}.json — the backend saying "here's where everyone should go next and what they're doing."

Advance the game clock by 10 seconds and increment the step counter.

The needs system Andus added

The original Stanford paper didn't have physical needs. Andus extended it so agents track hunger, energy, social drive, and more — and their plans react when those needs get low.

🍔 Hunger72%
⚡ Energy55%
👥 Social40%
💧 Hydration88%
reverie.py — tick_needs() 
for need, base_rate in s.needs_decay_rates.items():
  multiplier = get_persona_decay_multiplier(
    persona_name, need)
  rate = base_rate * multiplier

  if need == "energy":
    if is_sleeping:
      current_val += 1/60  # Restore during sleep
    else:
      current_val -= rate
  elif need == "social":
    if not is_chatting:  # Only decays alone
      current_val -= rate

  s.needs[need] = max(0, min(100, current_val))
Plain English

Every tick, loop through each need (hunger, energy, social, hydration, etc.).

Get a persona-specific multiplier — Maria might get hungry faster than Klaus.

Energy is special: it restores during sleep instead of decaying. The +1/60 means full recovery after 60 sleep ticks (~10 game minutes).

Social is also special: it only decays when the agent is alone. Having a conversation satisfies the need.

Clamp everything between 0 and 100 — needs can't go negative or exceed full.

05

How Agents Plan Their Day

From "wake up at 7am" to "walk to tile (58, 39)" — a three-layer planning system.

Planning happens in three layers

Think of it like a project manager breaking down a goal. The LLM is called separately at each layer — from broad strokes to specific actions.

📅
Layer 1: Daily Plan
"Wake up at 7am, eat breakfast, head to Oak Hill College by 9am, teach a philosophy lecture, have lunch at the diner at noon..."

Generated once per game day. ~5–8 broad events.
🕐
Layer 2: Hourly Plan
"9:00–9:10: walk to classroom. 9:10–10:00: lecture on epistemology. 10:00–10:15: answer student questions..."

Decomposed when the hour begins.
📍
Layer 3: Current Action
"the_ville:Oak Hill College:classroom:blackboard"

A specific address in the Maze. This is what gets passed to A* pathfinding.

The daily planning chain in code

plan.py — generate_first_daily_plan() 
def generate_first_daily_plan(persona,
    wake_up_hour, resource_manager=None):
  """
  Returns a list like:
  ['wake up and complete morning routine at 6am',
   'eat breakfast at 7am',
   'head to the cafe to work at 8am', ...]
  """

  # Build context from persona's identity + lifestyle
  return run_gpt_prompt_daily_plan(
    persona, wake_up_hour, resource_manager)[0]
Plain English

The function takes the persona (which carries their identity: name, traits, lifestyle, relationships) and their wake-up hour.

It also accepts the resource_manager — Andus's addition — so if the cafe is low on coffee, that context influences the plan.

It calls run_gpt_prompt_daily_plan() which builds a prompt with the persona's full identity and asks the LLM to write out their day.

Returns a list of broad string actions. This is the skeleton of the entire day.

What's in a prompt?

Every LLM call in this codebase follows the same pattern: build a string prompt from templates, call the API, parse the response. The prompts live in persona/prompt_template/. A typical planning prompt looks like:

📝
What the planning prompt actually contains

Name: Isabella Rodriguez
Traits: friendly, outgoing, passionate about art
Lifestyle: Isabella goes to Hobbs Cafe in the morning, heads to Oak Hill College at 10am...
Current date: Monday June 25, 2022
Recent memories: [top-k retrieved memories]

In broad strokes, write out Isabella's schedule for today.

The LLM doesn't "know" Isabella. It's given Isabella as context every single time. The memory system is what creates the illusion of continuity.

What happens when a plan changes

Agents can replan mid-day. If Isabella's hunger drops below 40, the needs system injects a perception event: "Isabella feels hungry and notices the kitchen nearby." This triggers a replanning check.

1
Need drops below threshold (e.g., hunger < 40%)

perceive_resource_context() generates a natural-language perception string.

2
Event injected into memory

The perception becomes a new event in AssociativeMemory with high poignancy — guaranteed to surface in retrieval.

3
LLM decides whether to replan

The plan module checks: "Given this new event, does Isabella need to change what she was doing?" If yes, generates new hourly plan.

4
New address → new path

A* pathfinding recalculates the route to the kitchen/cafe. The old plan is discarded.

💡
Why this matters for AI development

This is the pattern for all reactive AI systems: perception → memory injection → retrieval → LLM decision. The difference between a basic chatbot and a sophisticated agent is this feedback loop. Once you understand it here, you'll see it everywhere in agentic AI.
